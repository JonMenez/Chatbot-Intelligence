CHATBOT AI - RAG SYSTEM DOCUMENTATION
=====================================

## What is RAG?

RAG (Retrieval-Augmented Generation) is an AI architecture that combines two key components:

1. **Retrieval**: Search relevant documents or context from a knowledge base.
2. **Augmented Generation**: Use that retrieved context to generate more accurate, grounded responses.

Think of it like a MERN developer looking up documentation before writing an answer—instead of relying on memory alone (LLM hallucinations), the model has concrete references.

## How RAG Works in This Project

Our implementation uses:

- **Groq LLaMA 3.3 70B**: The language model (inference engine)
- **HuggingFace Transformers**: Local embeddings (Xenova/all-MiniLM-L6-v2)
- **MemoryVectorStore**: In-memory vector database for similarity search
- **LangChain.js**: Framework orchestrating the entire pipeline

### The RAG Pipeline (Step-by-Step)

1. **Document Loading**
   - Supports .txt and .pdf files
   - Files loaded from backend/documents folder
   - Validated for minimum content length (100 chars)

2. **Text Chunking**
   - Uses RecursiveCharacterTextSplitter
   - Chunk size: 600 characters (~150 tokens)
   - Overlap: 150 characters (25% overlap for context continuity)
   - Respects natural boundaries (paragraphs > sentences > words)

3. **Embedding**
   - Each chunk is converted to a vector (384-dimensional for all-MiniLM)
   - Done locally using Hugging Face Transformers (no API calls)
   - ~30-60 seconds for 100 documents on modern hardware

4. **Vector Storage**
   - Chunks and their embeddings stored in MemoryVectorStore
   - Currently in-memory (resets on server restart)
   - Future: Persist to SQLite or Supabase

5. **Retrieval (Query Time)**
   - User asks a question
   - Question is embedded using same model
   - Top-4 similar chunks retrieved via cosine similarity
   - Context assembled and sent to Groq

6. **Response Generation**
   - Groq LLaMA model receives system prompt + context + question
   - System prompt includes guardrails to prevent hallucinations
   - Model generates response grounded in retrieved context
   - Response returned with confidence score and source attribution

## Groq: The LLM Backbone

Groq is:
- **Fast**: Sub-second inference for large models
- **Free tier**: Generous rate limits for development
- **OpenAI-compatible**: Drop-in replacement for OpenAI API

We use the `llama-3.3-70b-versatile` model:
- 70 billion parameters
- Excellent for reasoning and code
- Good knowledge cut-off (training data)
- Temperature set to 0.1 for factual responses

### Groq API Configuration

```
Model: llama-3.3-70b-versatile
Temperature: 0.1 (low = more factual, less creative)
Max tokens: 800 (limits response length for conciseness)
Top-p: 0.9 (nucleus sampling for quality)
Timeout: 20 seconds
Retries: Automatic retry with exponential backoff
```

## HuggingFace Transformers: Local Embeddings

Why HF instead of OpenAI embeddings API?

- **Cost**: FREE (runs on your machine)
- **Privacy**: Embeddings computed locally, never sent to external API
- **Reliability**: No API rate limits on embeddings
- **Speed**: After initial model download (~100MB), embeddings are instant

Model: `Xenova/all-MiniLM-L6-v2`
- 384-dimensional embeddings
- 66 million parameters
- Optimized for semantic similarity
- Multilingual support

## Key Improvements in This RAG System

### 1. Hallucination Prevention
- System prompt explicitly forbids inventing answers
- Instructions: "If information is not in context, say: 'I don't have enough information...'"
- Temperature lowered to 0.1 for factuality over creativity

### 2. Better Text Chunking
- Increased overlap from 100 to 150 chars (better context continuity)
- Reduced chunk size from 800 to 600 chars (more granular)
- Respects document structure (paragraphs > sentences > words)

### 3. Rich Observability
- Logs which documents were retrieved
- Tracks retrieval latency and LLM latency
- Returns confidence score with each response
- Source attribution for transparency

### 4. Production-Ready Error Handling
- Validates document length (skip empty/tiny files)
- Retry logic with exponential backoff for rate limits
- Graceful degradation (server stays up if RAG init fails)

## Using the /rag Endpoint

### Request Format
```json
{
  "message": "Explain how RAG prevents hallucinations"
}
```

### Response Format
```json
{
  "reply": "RAG prevents hallucinations by...",
  "confidence": 0.85,
  "sources": ["info.txt"],
  "metadata": {
    "retrievalTime": 42,
    "llmTime": 1200,
    "totalTime": 1242,
    "documentsRetrieved": 4,
    "modelUsed": "llama-3.3-70b-versatile",
    "confidenceReasoning": "Retrieved 4 relevant documents"
  }
}
```

### Response Fields
- **reply**: The actual answer from the model
- **confidence**: Score 0-1 (based on retrieval quality)
- **sources**: Which documents were used
- **metadata**: Performance metrics and debugging info

## Common Issues & Solutions

### Issue: "RAG initialized with 0 chunks"
**Cause**: No documents in backend/documents folder or all files too small.
**Solution**: Add .txt or .pdf files to backend/documents, ensure > 100 characters.

### Issue: Slow startup (30+ seconds)
**Cause**: First time loading HF Transformers model (~100MB download + initialization).
**Solution**: Normal on first startup. Subsequent restarts are faster (~5-10s for inference cache).

### Issue: Response is "I don't have enough information"
**Cause**: Query too different from document content or context too small.
**Solution**: Add more documents or ask more specific questions matching your knowledge base.

### Issue: Groq 429 Rate Limit
**Cause**: Too many requests in short time.
**Solution**: Built-in retry with exponential backoff. Check free tier limits on Groq console.
